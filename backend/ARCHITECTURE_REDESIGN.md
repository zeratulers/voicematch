# VoiceMatch 架构重新设计文档

## 概述

根据您的需求，我们重新设计了系统架构，以实现以下业务逻辑：

1. **指令库管理**：医生可以创建和管理不绑定特定患者的指令
2. **方言变体管理**：每个指令可以有多种方言变体
3. **患者指令分配**：将特定的方言变体分配给患者
4. **模板系统**：管理员可以创建默认指令库和变体

## 新的数据库架构

### 核心表结构

#### 1. commands（指令库）
- `id`: 指令UUID
- `content`: 指令内容明文
- `description`: 指令描述
- `doctor_id`: 创建医生ID
- `is_active`: 是否激活
- `is_template`: 是否为系统模板
- 不再直接绑定患者

#### 2. command_variants（指令变体）
- `id`: 变体ID
- `command_id`: 所属指令ID
- `dialect_set_id`: 方言类型ID
- `audio_url`: 音频文件路径
- `speaker_name`: 说话人信息
- `is_template`: 是否为系统模板

#### 3. patient_command_assignments（患者分配）
- `id`: 分配ID
- `patient_id`: 患者ID
- `command_id`: 指令ID
- `variant_id`: 使用的变体ID
- `assigned_by`: 分配人（医生）
- `is_active`: 是否激活

#### 4. dialect_sets（方言集）
- `id`: 方言ID
- `key`: 方言代码（zh-cn, zh-yue等）
- `label`: 方言显示名称
- `is_default`: 是否为默认方言

## 新的API架构

### 指令管理 API (`/api/v1/commands`)

- `GET /` - 获取指令库列表（支持过滤模板、搜索等）
- `POST /` - 创建新指令
- `GET /{id}` - 获取指令详情（包含所有变体）
- `PUT /{id}` - 更新指令
- `DELETE /{id}` - 删除指令（软删除）
- `GET /stats/doctor` - 获取医生统计信息

### 患者分配 API (`/api/v1/assignments`)

- `GET /patient/{patient_id}` - 获取患者的指令分配
- `POST /` - 创建单个分配
- `PUT /{assignment_id}` - 更新分配
- `DELETE /{assignment_id}` - 删除分配
- `POST /batch` - 批量分配指令

### 指令变体 API (`/api/v1/commands/{command_id}/variants`)

- `GET /` - 获取指令的所有变体
- `POST /` - 为指令创建新变体
- `PUT /{variant_id}` - 更新变体
- `DELETE /{variant_id}` - 删除变体

## 业务流程

### 医生工作流程

1. **术前准备阶段**：
   - 在指令库中创建或管理指令
   - 为每个指令录制/上传不同方言的变体
   - 管理变体的激活状态

2. **患者准备阶段**：
   - 创建患者记录
   - 根据患者方言需求，将合适的指令变体分配给患者
   - 可以批量分配多个指令

3. **术中使用阶段**：
   - 系统根据患者分配播放对应的方言变体
   - 可以实时激活/停用特定分配

### 管理员工作流程

1. **系统模板管理**：
   - 创建系统默认指令（`is_template=true`）
   - 为模板指令创建默认变体
   - 新医生注册时自动继承这些模板

2. **方言管理**：
   - 管理系统支持的方言类型
   - 设置默认方言

## 权限控制

### 医生权限
- 可以查看和使用所有模板指令
- 只能修改自己创建的指令
- 只能为自己的患者分配指令
- 可以查看指令的患者使用统计

### 管理员权限
- 可以管理所有指令和变体
- 可以创建和管理系统模板
- 可以查看全系统统计信息
- 可以管理方言设置
